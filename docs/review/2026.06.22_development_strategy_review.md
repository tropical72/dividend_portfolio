## 1. SDD / TDD 개발 및 품질 관리 프로세스 분석

프로젝트의 헌법이라 할 수 있는 [ANTIGRAVITY.md](file:///home/tropical72/Work/dividend_portfolio/ANTIGRAVITY.md)와 [GLOBAL_STANDARD.md](file:///home/tropical72/Work/dividend_portfolio/docs/00_RAMS_Global/GLOBAL_STANDARD.md)를 바탕으로 파악한 프로세스 통제 전략은 다음과 같습니다.

### 1.1. SDD Quad-Update Rule (명세 기반 개발)
요구사항이나 기능의 변경이 발생할 때, 코드 작성 전 반드시 아래 4종의 문서를 정밀하게 동시 업데이트하여 문서와 코드 간의 100% 정합성을 보장하는 규칙입니다.
1. [trace_matrix.md](file:///home/tropical72/Work/dividend_portfolio/docs/trace_matrix.md): 전체 태스크 ID 부여 및 상태 추적용 마스터 시트.
2. `docs/{Domain}/requirement.md`: 도메인별 상세 비즈니스/시스템 요구사항 명세.
3. `docs/{Domain}/plan.md`: 구현 일정 및 Micro-Task 설계서.
4. `docs/{Domain}/test.md`: 자동/수동 테스트 시나리오 정의서.

> [!NOTE]
> 설계서와 개발 산출물 간의 격차(Gap)를 줄이기 위한 가장 성숙한 설계 통제 기법 중 하나입니다. 코드 리뷰 시 이 4개 문서가 최신 구현과 일치하는지가 최종 품질 승인의 핵심 척도가 됩니다.

### 1.2. TDD (Test-First Approach) 및 Micro-Task
- **선(先) 테스트 작성:** 코드를 변경하기 전에 반드시 실패(Red)하는 테스트 코드를 먼저 작성하여 요구 사양을 코드로 고정하고, 구현 후 성공(Green)을 확인합니다.
- **Atomic Work Unit:** 테스트-구현-Linting-커밋이 하나의 초소형 작업 단위(Micro-Task)로 쪼개져 진행됩니다.
- **외부 API Mocking & DB 격리:** 테스트 실행 시 `yfinance` 등 외부 API 요청은 철저히 모킹하며, E2E 테스트 실행 전 백엔드 상태를 스냅샷하고 종료 후 복원(/api/test/state)하여 실제 데이터 오염을 차단합니다.

### 1.3. Quality Gate (정적/동적 검증 강제)
커밋 전에 아래의 게이트를 강제 통과해야 합니다.
- **Python:** `ruff check`, `black --check`, `pytest`
- **Frontend:** `npm run lint`, `npm run build`, `prettier --check`, `Playwright` E2E 테스트

## 2. 도메인별 설계 사양 및 알고리즘 전략 검토

### 2.1. System Core & Global Standard (00_RAMS_Global)
- **수익률 지표의 표준화:** 자산의 총수익률(TR)을 배당수익률(DY)과 기대주가상승률(PA)의 단순 합(`TR = DY + PA`)으로 정의합니다. 포트폴리오 매니저는 DY만 계산하고, PA는 Settings에 등록된 자산군별 시장 평균 성장률을 전역 적용합니다.
- **수익률 월 변환 정밀화:** 장기 시뮬레이션의 복리 효과를 엄밀히 반영하기 위해 PA의 월 변환 시 단순 12분할이 아닌 복리 계산(`월 PA = (1 + 연 PA)^(1/12) - 1`)을 수행하고, 분배금(DY)은 현금 인출용 수확을 위해 `연 DY / 12`로 처리합니다.
- **자산군별 PA 시나리오 3중화:** `SGOV Buffer`, `Bond Buffer`, `High Income`, `Dividend Growth`, `Growth Engine` 5개 자산군에 대해 `conservative / base / optimistic` 3가지 시나리오를 설정하여 투자 성향별 프로젝션이 가능하게 합니다.
- **환경 분리:** 공개 저장소의 기본값(`defaults/`)하고 실제 운영 환경의 사용자 로컬 데이터(`APP_DATA_DIR` 우선)를 격리하여 보안성과 이식성을 동시에 확보합니다.

### 2.2. 은퇴 엔진 (01_Retirement_Engine)
- **5단계 전략 카테고리 구조:** 자산군을 `SGOV Buffer(초단기 안전)`, `Bond Buffer(미국 중기 국채)`, `High Income(고배당)`, `Dividend Growth(대형주 코어)`, `Growth Engine(성장 가속)`으로 명확히 구분하여 각각 독립된 버킷과 성장률/하한선 규칙을 적용합니다.
- **SGOV-Only 월 인출 원칙:** 매월 필요한 가계 생활비 및 법인 비용은 오직 현금성 자산인 `SGOV Buffer`에서만 인출하고, 다른 자산군은 정기점검 시에만 donor 규칙에 따라 매도하여 버퍼를 보충합니다. 이는 하락장에서 성장형 자산의 강제 매도를 방어(Bear Market Freeze)하기 위한 핵심 알고리즘입니다.
- **정기/비정기 운영 캘린더:**
  - **5월(메인 정비):** 법인 SGOV 30개월 복구, 개인연금 SGOV 24개월 복구, VGIT/주식 비중 재조정.
  - **8월(미니 점검):** 법인세 중간예납 등 확정 현금이벤트 반영 후 SGOV만 조정(슬리브 리밸런싱 금지).
  - **11월(반기 정비):** 법인 SGOV 27개월 복구 및 채권 상태 점검.
- **인플레이션 판정 및 Stress State:**
  - 5월 정기점검 시점에 법인 SGOV 30개월 재구축 여부, 법인 VGIT 12개월 floor 유지 여부, 총 운영자산의 실질 가치 유지 여부(A/B/C 테스트)를 검사하여 하나라도 실패하면 `Stress` 상태로 전환합니다.
  - `Stress` 상태에서는 인플레이션 반영이 동결되어 생활비 인상이 제한됩니다.
- **Shock Event & BOOST:**
  - 월말 주식성 슬리브(VOO, QQQM 등) 평가액이 직전 5월의 80% 이하로 급락하면 `Crash20 Shock Flag = ON`이 됩니다.
  - 이 기간 동안 개인연금은 법인의 주식 저가매도 강제를 피하기 위해 월 100만~300만 원의 `BOOST` 인출을 한시(6개월) 증액 실행합니다.
- **세무/비용 결합:**
  - 법인세 과세표준은 미실현 PA를 제외한 배당/이자/인컴 등 `실현소득` 기준으로 계산합니다.
  - 가계 세후 필요금액(`household_monthly_need`)에서 연금 실수령액과 급여 실수령액을 차감한 **세후 부족분만큼만 주주대여금 상환**을 실행함으로써 비과세 한도(`initial_shareholder_loan`)를 효율적으로 소진합니다.
  - 개인 일반계좌에 대한 미국 배당 원천세(지급월), 종합소득세 및 양도소득세(다음 해 5월 납부), 지역건강보험료 시차(재산세 과세표준 반영 및 금융소득 반영 시차)를 정밀하게 반영하고 연도별 감사표(`personal_annual_tax_audit`)를 출력합니다.

### 2.3. 비용 비교 시뮬레이터 (02_Cost_Comparison_Simulator)
- **개인 vs 법인 껍데기 운용 비교:** 동일 포트폴리오, 동일 TR 하에서 법인세/소득세, 건강보험료(지역 vs 직장), 운영비용 차이에 따른 경제적 실익을 검증합니다.
- **비교 모드 이원화:**
  - **목표 수익 중심 (Target-driven):** 가구 목표 실수령을 달성하기 위한 비용 비교.
  - **보유 자산 중심 (Asset-driven):** 고정 자산의 세후 현금흐름 및 비용 누수율 비교.
- **건강보험료 정밀 시뮬레이션:** 지역건보료 계산 시 단순 공식이 아닌 공단 기준의 재산 60등급표 및 소득월액보험료 공식, 장기요양보험료율을 그대로 반영하여 결과의 정합성을 확보합니다.
- **우열 판정 기준:** 가계 이전 경로(주주대여금 상환, 급여 등)가 아닌 최종 세후 `순현금흐름(net cashflow)`을 기준으로 승자를 판정합니다.

## 3. SW Master 종합 평가 및 품질 확정 전략

### 3.1. 강점 (Strengths)
1. **아키텍처의 격리성:** 데이터 스키마가 설정, 포트폴리오 데이터, 시나리오 가정을 명확히 분리하고 있어 비즈니스 변경에 유연합니다.
2. **엄격한 TDD 프로세스:** `test_retirement_os_v11_1_engine.py` 등 상세한 테스트 커버리지를 보장하는 설계 파일들이 구비되어 있어 안정적인 리팩토링이 가능합니다.
3. **현실적인 도메인 모델링:** 건강보험료의 부과 등급제나 법인세 중간예납/확정 세액 계산의 시차 반영 등 현업 세무 규칙을 높은 완성도로 모델링했습니다.

### 3.2. 검수 주의점 (Quality Checklist for Code Review)
향후 코드를 전면 검수할 때 다음과 같은 잠재 위험 요소를 집중 체크할 예정입니다.

1. **실현/미실현 수익 계산 정합성:**
   - 자산 평가액 증가(PA)가 법인세 과세표준이나 개인 종합소득세 대상에 누출되어 합산되는 코드가 없는지 철저히 확인해야 합니다.
2. **SGOV Only 인출 규칙 강제성:**
   - `projection_engine.py` 내의 월간 인출 로직에서 SGOV 이외의 버킷 잔고가 직접 차감되는 누수 현상이 없는지 검사해야 합니다.
3. **분배금 Run-Rate 및 리밸런싱 동기화:**
   - 카테고리 간 자산 이동 시 run-rate가 `이동 금액 / 이전 평가액` 비율대로 정확히 가감되는지, 신규 매수 시 target DY를 기준으로 run-rate가 정확하게 생성되는지 수학적 정합성을 확인해야 합니다.
4. **메인 정기점검 월의 동적 오프셋 적용:**
   - `rebalance_month`가 5월이 아닌 임의의 월로 변경되었을 때, 8월 미니점검(+3개월), 11월 반기점검(+6개월) 캘린더 연산과 8월 중간예납/3월 확정 법인세 캘린더가 꼬이지 않고 올바른 인덱스(Month)에 트리거되는지 수식 정합성을 검증해야 합니다.
